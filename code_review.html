Bugs

Bug Help file only opens on Windows L1269
os.startfile() is Windows-only. On macOS or Linux it silently does nothing — no error, no fallback. Since the README mentions Linux Tkinter setup, this is a real gap. 
# Current:
os.startfile(help_path) if os.name == 'nt' else None

# Fix:
import subprocess, sys
if os.name == 'nt':
    os.startfile(help_path)
elif sys.platform == 'darwin':
    subprocess.Popen(['open', help_path])
else:
    subprocess.Popen(['xdg-open', help_path])
Bug Silent duplicate try/except in add_row_from_inputs L1052–1055
Both the try and except branch do the exact same thing — writing the cell value. The error is swallowed without logging, meaning if the write fails you'll never know. 
# Current (both branches identical):
try:
    sheet.cell(row=append_row_excel, column=col_index_excel).value = parsed
except Exception:
    sheet.cell(row=append_row_excel, column=col_index_excel).value = parsed

# Fix: just write it, and if you must catch, log it:
try:
    sheet.cell(row=append_row_excel, column=col_index_excel).value = parsed
except Exception as e:
    self._update_status(f"Cell write error col {col_index_excel}: {e}", "error")
Bug Shadow row index desync after deletion L1198–1212
When a row is deleted from the middle of the tree, the shadow_values dict retains its old row indices. Any subsequent hover or edit on rows below the deleted one will read stale or mismatched shadow data. You need to re-index shadow keys after deletion. 
# After deleting excel_row_index, shift all shadow keys:
new_shadow = {}
for (sname, ridx, cidx), val in self.shadow_values.items():
    if sname == self.active_sheet_name and ridx > display_row_index:
        new_shadow[(sname, ridx - 1, cidx)] = val
    elif ridx != display_row_index:
        new_shadow[(sname, ridx, cidx)] = val
self.shadow_values = new_shadow
Bug Ctrl+N is new file, not "clear fields" as documented L370 vs README
The README says Ctrl+N = "Clear input fields", but the code binds it to self.new_file(). The shortcut for clearing inputs (reset_to_add_mode) is Escape. One of them needs to be corrected — either the docs or the binding.

Warnings & Code Quality

Warning Bare except: clauses throughout Multiple
Over 20 bare except: or except Exception: pass blocks swallow errors silently. This makes debugging extremely hard. You should at minimum log to a status bar or console. Consider a simple debug flag: 
DEBUG = os.getenv("APP_DEBUG", "0") == "1"

def _safe(self, fn, *args, label="", **kwargs):
    try:
        return fn(*args, **kwargs)
    except Exception as e:
        if DEBUG:
            print(f"[{label}] {e}")
        return None
Warning Two overlapping tooltip systems L35–61, L300–327
There is a standalone ToolTip class AND a manual _show_tooltip / _hide_tooltip method inside the app class doing the same thing. The class-based one binds to Enter/Leave; the method-based one is called from hover logic. Pick one and delete the other.
Warning _flush_shadow_to_workbook is a stub that does nothing L1217–1219
The method exists with a comment "safety flush if needed" but has only a pass body. Either implement it or remove it. Dead code like this misleads future readers.
Warning Prefs file saved next to the Excel file (pollution risk) L1281
filepath + ".prefs.json" creates a config file in the user's spreadsheet directory, which may be a shared or cloud-synced folder. A cleaner approach is a standard app-config directory: 
import platformdirs
prefs_dir = platformdirs.user_config_dir("tEppy_DataEntry")
os.makedirs(prefs_dir, exist_ok=True)
# Use a hash of filepath as the prefs filename

Strengths 🏅

Praise Excel-style rounding with Decimal is excellent L178–186
Using Decimal + ROUND_HALF_UP avoids Python float rounding quirks. This is exactly how Excel rounds, and using the "▲ " prefix to visually flag rounded values in the tree is a clever, user-friendly touch.
Praise Shadow value store is a smart design L285–295
Keeping a parallel dict of raw/parsed/rounded data separate from the Treeview display is architecturally sound. Treeviews can only store strings, so this pattern cleanly separates display from data — good thinking.
Praise Universal file loader handles 5 formats gracefully L66–103
load_any_excel() is well-structured with proper extension routing, embedded image warnings, and a clean fallback to in-memory conversion. The RuntimeError wrapping gives useful messages to the UI layer.

Current Structure (Everything in one file)

📄 aaa_gemini.py — 1,349 lines, all logic in one place
The app is a monolith: utilities, UI, file I/O, validation, business logic, and state management are all inside one giant class + a few top-level functions. For a solo project this is workable, but as features grow (and they will), it becomes increasingly painful to navigate and test.

✅ Recommended Split (4 files)

📁 data_entry_app/
📄 main.py — entry point, Window init only
📄 app.py — DynamicExcelApp class, UI construction, event wiring
📄 excel_io.py — load_any_excel, save logic, prefs read/write
📄 validation.py — all validators, normalizers, rounding, date parsing
📄 widgets.py — ToolTip, filter row, shadow store helper class
📄 help.txt
Benefits: You can unit-test validation.py in isolation (pytest, no GUI needed). excel_io.py can be reused or swapped. The UI code in app.py becomes much shorter and readable — probably ~400 lines instead of 1,000+.

State Management Concern

DynamicExcelApp.__init__ accumulates 15+ instance attributes that are interdependent: workbook, filepath, active_sheet_name, current_sheet, headers, validation_rules, current_rules, input_order, shadow_values, mode, editing_item, original_editing_values…

Consider grouping related state into small dataclasses: 
from dataclasses import dataclass, field

@dataclass
class AppState:
    workbook: object = None
    filepath: str = None
    active_sheet_name: str = None
    headers: list = field(default_factory=list)
    mode: str = "add"
    unsaved_changes: bool = False
This makes it much easier to reset state cleanly when opening a new file.

What the Code Does Well (Architecturally)

The separation between display layer (Treeview with string values) and data layer (shadow dict + openpyxl workbook) is the right instinct. The validation rules being stored as dicts with BooleanVar/StringVar tied to UI controls is also clean — the UI and logic share the same data structure without a complex event bus.

High-Value Additions to Consider

Feature Idea Undo / Redo (Ctrl+Z / Ctrl+Y)
Currently there's no way to undo an accidental add or edit. A simple command history stack using Python's collections.deque with snapshots of the workbook sheet rows would cover the most common use cases. Even just a single-level undo ("undo last add") would be a huge quality-of-life improvement.
Feature Idea Row count & column summary in status bar
Show "Rows: 142 · Filtered: 12 · Sheet: Sales" in the status bar. This is a tiny change but gives immediate data context, especially when filters are active and the user can't tell how many rows are hidden.
Feature Idea Duplicate Row shortcut (Ctrl+D)
The README documents Ctrl+D as "Duplicate row" but there is no binding or method for it in the code. This is an unimplemented feature — add it as a quick win: copy selected row values into the input fields without committing, letting the user tweak before adding.
Feature Idea Export filtered view to new Excel file
When the user has filtered the treeview to a subset of rows, add a "Export View" button/menu item that saves only the currently visible rows as a new .xlsx. Very useful for office clerks who need to share a subset of the master sheet.
Feature Idea Column sort by clicking headers
Clicking a Treeview column heading could toggle sort ascending/descending. Tkinter Treeview doesn't do this automatically, but it's straightforward to implement with an all_rows re-sort and tree reload. Add a sort indicator (↑ ↓) to the heading text.
Feature Idea Optional dark/light mode persistence per-theme
The theme is saved per-file in prefs. Consider making it a global app preference instead — most users want the same theme across all files. Move it to a separate app_global.prefs.json in the user config dir.
Quick Win Ctrl+D (Duplicate) is documented but missing
Add to _bind_shortcuts: 
self.root.bind("", lambda e: self.duplicate_selected_row())

def duplicate_selected_row(self):
    sel = self.tree.focus()
    if not sel: return
    values = self.tree.item(sel, "values")
    self.clear_input_entries()
    for i, entry in enumerate(self.input_entries):
        if i < len(values):
            v = values[i]
            if isinstance(v, str) and v.startswith("▲ "):
                v = v[2:]
            entry.insert(0, v)
    self.reset_to_add_mode()
    self._update_status("Row duplicated into input fields.", "info")
Quick Win Insert Blank Row (Ctrl+Shift+I) also missing
Like Ctrl+D, this is documented in the README but not implemented in the current code file. It would insert an empty row in the workbook at the current selection index using sheet.insert_rows().
